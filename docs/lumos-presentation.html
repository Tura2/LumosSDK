Conversation Tracing
Record every prompt, response, model, tokens, and latency. Full trace lifecycle — start to end.
Offline-First Queue
Events stored in Room (SQLite). WorkManager uploads in batches of 50 when network is available.

Developer Interface
Public API Functions
Setup & Tracing
// 1. Init once in Application.onCreate()
LumosSDK.init(context) {
  apiKey = "lms_xxxx"
  serverUrl = "https://api.lumos.dev"
}

// 2. Start a trace per AI conversation
val trace = LumosSDK.startTrace("chat")

// 3–4. Log input and AI response
trace.logPrompt("What is Kotlin?")
trace.logResponse(text, model, tokensIn, tokensOut, latencyMs)

// 5. Track a sub-step (tool call, RAG...)
val span = trace.startSpan("web_search")

// 6–7. Error handling and close
trace.logError(exception)
trace.end(TraceStatus.OK)

Why These Choices?
Pre-aggregated stats_hourly
Dashboard queries hit a tiny summary table — O(hours) not O(millions of traces). 100× faster at scale, updated on every event ingestion.
Keyset Pagination
WHERE id < cursor instead of OFFSET. Stays fast at 1M+ rows where OFFSET degrades linearly.
Deduplication Table
SDK retries are safe — ingested_events tracks processed event IDs. System is fully idempotent.

Technical Research
Deep Dive: Key Findings
Android Telemetry — What's Safe
  • Safe to collect: OS version, device model, network type (WiFi/Cell/None), app version — via Build class & ConnectivityManager
  • Never collect: GPS, IMEI, contacts, camera — avoids PII and permission requirements
  • Our telemetry set requires zero Android permissions
Crash Capture Before App Closes
  • Hook: Thread.defaultUncaughtExceptionHandler
  • Must be synchronous — coroutines won't run after JVM crash is initiated
  • Write to Room synchronously → mark trace ERROR → re-throw to let OS handle the crash
synchronous flush only
Battery & Network Optimization
  • WorkManager respects Doze mode & App Standby — no wakelocks, no ANRs
  • Batching 50 events = 50× fewer HTTP connections vs per-event upload
  • Gzip compression ≈ 80% size reduction → less radio-on time, less battery drain
50x fewer connections
Analytics Architecture Decisions
  • Pre-aggregation: stats_hourly updated per ingestion → O(1) dashboard query instead of a full table scan
  • Keyset pagination: cursor-based avoids OFFSET — stays fast at 1M+ rows where OFFSET degrades linearly
  • Considered full OLAP scans — rejected due to latency at scale on a single SQLite file
100x faster dashboard queries
